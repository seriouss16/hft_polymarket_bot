# План улучшений LIVE / CLOB (Polymarket)

Документ раскладывает целевую архитектуру (WS-first, FSM, анти-гонки, partial fill) на **этапы**, которые можно внедрять итеративно без «большого взрыва». Кодовая база: `live_engine.py`, `clob_user_ws.py`, `clob_market_ws.py`, основной цикл в `bot_main_loop.py`.

---

## Цели

- Единая **последовательная** обработка событий исполнения (очередь + FSM), чтобы убрать race между stale/reprice/cancel/fill.
- **WS как основной источник** состояния ордеров и fill; REST — старт, реконнект, reconcile.
- Явные правила **partial fill**, без «emergency BUY» поверх уже исполняющегося ордера без сверки.
- Контроль **свежести стакана** перед размещением/перестановкой.
- Согласованная терминология **maker/taker** (bid/ask) в коде и логах.

---

## Этап 0 — Согласование спеки и метрик

**Содержание**

- Зафиксировать в коде/конфиге единицы: спред в **тиках** (0.01) или в долях; пороги stale в **мс** (профили DAY/NIGHT).
- Развести понятия: **покупка maker** (лимит у bid) vs **покупка taker** (лимит ≥ best ask); то же для продажи.
- Добавить счётчики/лог-коды для диагностики: `STALE_ORDERBOOK`, `SPREAD_TOO_WIDE`, `PARTIAL_FILL`, `OUT_OF_ORDER_EVENT`, `WS_RECONNECT`, `REST_RECONCILE`.

**Критерий готовности**

- Таблица env-параметров и лог-событий описана; нет противоречий «покупка = bid» при сценарии «ставим ask».

**Риски**

- Минимальные (документация + пара лог-строк).

---

## Этап 1 — Свежесть данных и запрет действий

**Содержание**

- Перед `place` / `reprice` / `cancel` (кроме аварийного flatten): проверка `is_fresh()` по рыночному WS и при необходимости по user WS.
- Пороги вынести в конфиг (например агрессивный 200 ms — только если метрики показывают, что не режет сделки впустую).
- При stale: не инициировать новую цепочку исполнения; опционально только `REST` сверка позиции/ордеров.

**Критерий готовности**

- В логах видно явное `SKIP: stale_orderbook` с причиной и возрастом.

**Связь с кодом**

- `clob_market_ws.py`, существующие `CLOB_*_STALE_*`; вызовы из слоя live перед исполнением.

---

## Этап 2 — Heartbeat, реконнект, REST-сверка

**Содержание**

- Периодический heartbeat (интервал из конфига, напр. 4.5 s) по user/market WS: при отсутствии сообщений — реконнект + **REST sync** (открытые ордера, баланс conditional, критичные поля).
- После реконнекта: ресабскрайб, однократная сверка состояния FSM с REST.

**Критерий готовности**

- Нет «тихого» зависания: таймауты на I/O остаются (запрет — только `sleep` в горячем пути **решений**, не на сетевых операциях).

**Риски**

- Частые реконнекты при агрессивном heartbeat — калибровать по `msg_rate` / `last_msg_age` (уже есть зачатки в логах).

---

## Этап 3 — Очередь событий и FSM исполнения

**Содержание**

- Все события, влияющие на ордер: **WS** (placement, update, cancel, trade), **ответы REST** (place/cancel), **таймеры** (deadline ожидания подтверждения) — в **одну очередь**, обработка **строго одним воркером** (или asyncio lock + очередь).
- Состояния минимум: `IDLE`, `PLACING`, `OPEN`, `PARTIAL`, `CANCELING`, `FILLED`, `EXITING` (расширять по мере внедрения).
- Правило: не начинать `CANCEL`/`REPLACE`, пока не зафиксирован переход из `PLACING` (ACK от биржи или REST fallback по таймауту).

**Критерий готовности**

- Воспроизводимый сценарий из лога (stale + partial + emergency) не приводит к двойному действию на одном `order_id`.

**Связь с кодом**

- Рефакторинг `live_engine.py`: выделить машину состояний от «сырых» колбэков `clob_user_ws.py`.

---

## Этап 4 — Политика цены: узкий / широкий спред

**Содержание**

- Вынести в одну функцию расчёт лимита для **BUY** и **SELL** по best bid/ask и режиму (maker vs taker, «+1 тик»).
- Узкий спред: taker-покупка на best ask (или ask+1 тик по правилам тика); широкий: постановка у best bid+1 тик (maker) с таймаутом на дозаполнение.
- Продажа: симметрично (hit bid vs improve ask).

**Критерий готовности**

- Юнит-тесты на расчёт цены при фиктивной книге (без сети).

---

## Этап 5 — Partial fill и emergency

**Содержание**

- При `PARTIAL`: отмена **остатка** только после решения FSM; новый ордер на недобор — с новым `order_id`, после подтверждения cancel или по REST «остаток 0».
- Если сигнал/тренд «сломан»: при малом остатке — агрессивный выход (уже есть FAK-логика — увязать с FSM).
- Убрать или изолировать путь «emergency BUY» так, чтобы он **не** конкурировал с ордером, по которому ещё идут fill-события (сначала reconcile).

**Критерий готовности**

- Нет `PolyApiException` «not enough balance» из-за двойного резервирования одного и того же USDC на две заявки в одном окне.

---

## Этап 6 — Out-of-order и идемпотентность

**Содержание**

- Для каждого `order_id` хранить последний известный **sequence** / timestamp из WS (если доступно в payload); игнорировать «откаты» состояния назад без явного `canceled`.
- Если sequence нет — монотонное правило: `filled_size` только не убывает; `status` переходы по допустимой диаграмме.

**Критерий готовности**

- Тесты с подачей событий в обратном порядке.

---

## Этап 7 — Задержки: горячий путь

**Содержание**

- Логирование в горячем пути — либо буфер/async sink, либо уровень INFO только для редких событий.
- Индикаторы (RSI/ADX/…) уже частично инкрементальные — гарантировать, что live-тик **не** пересчитывает тяжёлое при каждом шаге исполнения.
- Разделение: рыночные данные async; **решения по ордеру** — только из очереди FSM.

**Критерий готовности**

- Профилирование одного тика: нет лишних синхронных HTTP в main decision path.

---

## Этап 8 — Запреты и риск-правила

**Содержание**

- Явный запрет повторного входа, пока позиция/ордер в `UNKNOWN` (после reconcile → `IDLE` или `HEDGED`).
- Запрет усреднения вниз при сломанном тренде (политика уже близка к engine — формализовать в одном месте).
- Не отключать **таймауты** на сеть; запрет — на произвольный `sleep` в логике стратегии между зависимыми шагами без события.

## Ограничения ожиданий

Даже после полного прохождения этапов система остаётся **высокоскоростной ритейл-исполнительницей** в рамках Polymarket API, а не колокационным HFT в индустриальном смысле. Выигрыш — в **консистентности, отсутствии самоинфицированных убытков** и предсказуемом поведении при плохом WS.
